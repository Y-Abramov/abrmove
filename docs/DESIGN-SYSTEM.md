# ABR|MOVE — правила редактирования и создания страниц

Свод правил дизайн-системы сайта. Перед правкой существующих страниц или созданием новых — сверяться сюда. Источник направления: `docs/superpowers/` (спеки/планы), живая история решений — в памяти агента (`abr-move-design-status` и связанные).

## 1. Токены — единственный источник правды

Все цвета живут в `src/styles/theme.css`, объявлены дважды: `:root[data-theme="dark"]` и `:root[data-theme="light"]`.

- Никогда не хардкодить hex в компонентах — только `var(--fg)`, `var(--paint)`, `var(--muted)` и т.д.
- Новый цвет нужен сразу в обеих темах — добавляй пару значений одновременно, не только в dark.
- `global.css` не должен задавать цвет `html`/`body` напрямую — это ломает переключение темы (уже вычищено, не возвращать).

## 2. Типографика

- **Golos Text** — весь обычный текст: заголовки, body, кнопки, nav.
- **JetBrains Mono** (`.mono`) — только данные: версии, пикеты (ПК), хэши, аптайм-лейблы верхнего регистра (`.sec-k`, `.ver`, `.station`, бейджи). Никогда для целых предложений — так текст читается тяжелее.

## 3. Сигнатура — пикетный каркас

Разделители между крупными секциями главной — `<div class="datum">` со станционными отметками (`.station.l` / `.station.r`), не generic `<hr>`. Кодирует предметку (продольный профиль дороги), не декорация.

- Использовать только между секциями верхнего уровня главной (Hero / Modules / News).
- Не использовать внутри карточек, доков модуля или произвольных блоков — там своя структура (see §7, §8).

## 4. Акцентный цвет

- `--paint` (cyan в тёмной, deepened-cyan в светлой) — единственный интерактивный акцент: активные ссылки, кнопки, бейджи, hover-состояния, курсор демо.
- `--grade` (красный) зарезервирован под «проектный уклон» в демо плана/профиля. Не переиспользовать как error/danger-цвет форм без явного повода — переиспользование размоет предметный смысл.

## 5. Шапка и активное состояние nav (важно — так просил заказчик)

Визуал активной ссылки: `color: var(--paint)` + подчёркивание цветом paint, `text-underline-offset: 6px`, `text-decoration-thickness: 2px`. Класс `.on`. Портировано 1:1 из `public/sp42-2016-vs-2026.html` (`.navlinks a.active`).

Механизм — два уровня, оба уже реализованы, **не переизобретать**:

1. **Server-side**: `<Header active="home|modules|news" />` — статичный `.on` для страниц без якорных секций (например `/modules/[slug]`).
2. **Client scroll-spy** (`Layout.astro`, IIFE в конце body): слушает scroll, ищет секции по `id`, извлечённому из `href` (`#hash`), подсвечивает ту, что ближе к верху (`trigger = 60px шапка + 40px запас`), у низа страницы форсит последнюю ссылку.
3. Если добавляешь в nav новый якорь — секция обязана иметь совпадающий `id`, иначе scroll-spy её проигнорирует (и это ожидаемо — на страницах без секции просто останется server-side `.on`).

## 6. Как добавить новый модуль

Правь **только** `src/data/modules.ts` — добавь объект в массив `modules`. Всё остальное генерится автоматически:

- `slug` → роут `/modules/{slug}` (`pages/modules/[slug].astro`, `getStaticPaths`)
- карточка на главной (`Modules.astro`)
- пункт в левой навигации docs-страницы

Обязательные поля: `title`, `version`, `compat[]`, `compatShort`, `minVersion`, `short`, `lead`, `sections[]`, `faq[]`. Секции — html-строки; переиспользуй существующие классы (`.steps`, `.note`, `.code`) вместо новых, если смысл совпадает. `screenshot` / `gif` — путь в `/public`, опционально (плейсхолдер, если нет).

Источник правды по составу модулей — репозиторий `github.com/Y-Abramov/robur-modules` (`catalog.json`). Если каталог обновился — сверяй `modules.ts` с ним вручную (авто-синка нет).

## 7. Карточка модуля — «чертёжный штамп»

Фиксированная структура: `shot` (скрин/гифка/плейсхолдер) → `cbody` (заголовок+версия, описание) → `stamp` (`ABR·MOVE | совместимость | Подробнее →`). Не менять порядок — это сигнатурный паттерн, должен визуально рифмоваться с бейджами на странице модуля.

## 8. Страница модуля — 3 колонки

`pages/modules/[slug].astro` генерит все страницы из `modules.ts` — руками новые `.astro`-файлы под модули не создавать.

- **Левая** колонка — список модулей, сгруппирован по платформе. Группа `Autodesk Civil 3D` со значком `СКОРО` — задел под будущее, не удалять и не занулять, пока Civil-модулей нет.
- **Правая** — TOC, автогенерится из `sections` + `faq`. Свой собственный scroll-spy (внутри `[slug].astro`, слушает `.center h2[id]`) — не путать с nav-спаем в `Layout.astro`, это разные механизмы для разных целей.
- Нумерация секций (`01`, `02`…) — вычисляется по индексу массива, не хардкодить.

## 9. Новые самостоятельные (article-style) страницы

Прецедент — `public/sp42-2016-vs-2026.html`: цельный HTML-файл вне сборки Astro, с задублированными токенами/CSS/JS внутри самого файла. Это **legacy**, оставить как есть, не трогать без отдельного запроса.

**Правило на будущее**: новые длинные редакционные/статейные страницы (разборы норм, сравнения, статьи) делать как настоящие Astro-страницы в `src/pages/` — импортировать `Header`/`Footer` + `theme.css`, **не копипастить** токены в новый standalone HTML-файл. Даёт: тему из коробки, единый Header с nav+спаем, единый Footer, автообновление при правке `theme.css` вместо N копий токенов по файлам.

Если нужна «документная» типографика (плотные колонки, таблицы, как в sp42) — заводи классы в `theme.css` с уже принятым префиксом `.doc-*` (частично есть для module docs), не создавай параллельный CSS-файл.

## 10. Общее

- Весь user-facing текст — русский.
- Motion — сдержанный: `.rv` reveal, hover-микро, максимум один акцентный демо-момент на страницу. Не плодить анимации ради анимаций (см. память `design-iteration-pace-feedback` — источник «AI slop»-ощущения в прошлой версии сайта).
- Логотип — всегда пара `l-dark`/`l-light` (`/logo-lockup-white.png`, `/logo-lockup.png`), переключение чисто через CSS по `data-theme` (`display:none/block`), не JS-подменой `src`.
- Если новое требование заказчика противоречит правилу здесь — требование заказчика побеждает, но если решение окончательное (не разовый эксперимент) — обнови этот файл в том же заходе.
